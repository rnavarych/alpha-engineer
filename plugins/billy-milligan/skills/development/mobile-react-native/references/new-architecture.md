# React Native New Architecture

## Overview

```
Old Architecture:
  JS Thread <--Bridge (JSON serialization)--> Native Thread
  - Asynchronous, serialized communication
  - No direct memory sharing
  - Bridge bottleneck for frequent updates (animations, gestures)

New Architecture:
  JS Thread <--JSI (C++ interface)--> Native Thread
  - Synchronous, direct method calls
  - Shared memory via JSI
  - No JSON serialization overhead

Components:
  - JSI: JavaScript Interface — C++ layer replacing the bridge
  - Fabric: New rendering system — synchronous layout
  - TurboModules: Lazy-loaded native modules via JSI
  - Codegen: Type-safe interface generation from specs
```

## Fabric (New Renderer)

```
What changes:
  - Synchronous rendering — layout computed on both threads
  - Concurrent features — React 18 Suspense, Transitions
  - Better gesture handling — no async bridge delay
  - Shared C++ core — same layout engine as web (Yoga)

For app developers:
  - Mostly transparent — existing components work
  - Some third-party libs need updates (check compatibility)
  - Better performance for complex layouts and animations
```

```tsx
// No code changes needed for most apps
// Fabric is enabled by default in React Native 0.74+

// Verify Fabric is active:
import { Platform } from 'react-native';
console.log('Fabric enabled:', global.__fabric ?? false);
```

## TurboModules

```
What changes:
  - Native modules loaded lazily (on first use, not at startup)
  - Synchronous calls via JSI (no async bridge)
  - Type-safe specs generated by Codegen
  - Startup time improvement: modules not loaded until needed
```

```typescript
// TurboModule spec — defines the native interface
// specs/NativeCalculator.ts
import { TurboModuleRegistry, TurboModule } from 'react-native';

export interface Spec extends TurboModule {
  multiply(a: number, b: number): number;        // Synchronous!
  fetchData(url: string): Promise<string>;        // Async also supported
  getConstants(): { PI: number; E: number };
}

export default TurboModuleRegistry.getEnforcing<Spec>('Calculator');
```

```typescript
// Usage — same API, but now synchronous via JSI
import Calculator from './specs/NativeCalculator';

const result = Calculator.multiply(6, 7); // Synchronous — no bridge!
const pi = Calculator.getConstants().PI;
```

## JSI (JavaScript Interface)

```
JSI enables:
  - Direct C++ <-> JavaScript communication
  - No JSON serialization/deserialization
  - Synchronous method calls
  - Shared memory between JS and native

Libraries using JSI:
  - react-native-reanimated: UI thread animations
  - react-native-mmkv: synchronous storage
  - react-native-quick-sqlite: direct SQLite access
  - react-native-vision-camera: real-time frame processing
```

## Migration Guide

### Step 1: Check Compatibility

```bash
# Check if your dependencies support New Architecture
npx react-native-new-arch-check

# Common libraries and their status:
# react-navigation: supported
# react-native-reanimated: supported
# react-native-gesture-handler: supported
# react-native-screens: supported
# react-native-svg: supported (v13+)
```

### Step 2: Enable New Architecture

```json
// app.json (Expo)
{
  "expo": {
    "newArchEnabled": true  // Enables both Fabric and TurboModules
  }
}
```

```ruby
# ios/Podfile (bare React Native)
ENV['RCT_NEW_ARCH_ENABLED'] = '1'
```

```groovy
// android/gradle.properties (bare React Native)
newArchEnabled=true
```

### Step 3: Handle Breaking Changes

```typescript
// Old: requireNativeComponent (bridge)
import { requireNativeComponent } from 'react-native';
const MyView = requireNativeComponent('MyNativeView');

// New: Fabric component with codegenNativeComponent
import codegenNativeComponent from 'react-native/Libraries/Utilities/codegenNativeComponent';
const MyView = codegenNativeComponent<NativeProps>('MyNativeView');

// Most third-party libs handle this internally
// Only matters if you maintain custom native modules
```

### Step 4: Test

```bash
# Build and test with New Architecture enabled
# iOS
cd ios && RCT_NEW_ARCH_ENABLED=1 pod install && cd ..
npx react-native run-ios

# Android
npx react-native run-android

# Watch for:
# - Crash on startup → dependency not compatible
# - Layout issues → Fabric rendering difference
# - Missing module → TurboModule not registered
```

## Anti-Patterns
- Enabling New Arch without checking library compatibility — crashes at runtime
- Writing bridge-based native modules for new projects — use TurboModules
- Ignoring Codegen type specs — lose type safety between JS and native
- Not testing on both platforms after enabling — iOS/Android may differ

## Quick Reference
```
JSI: C++ interface replacing JSON bridge — synchronous, shared memory
Fabric: new renderer — synchronous layout, React 18 concurrent features
TurboModules: lazy-loaded native modules via JSI
Codegen: generates type-safe interfaces from specs
Enable: newArchEnabled: true in app.json (Expo)
Check: npx react-native-new-arch-check for dependency compatibility
Default: React Native 0.76+ — New Architecture enabled by default
Migration: mostly transparent for app code, check third-party libs
```
